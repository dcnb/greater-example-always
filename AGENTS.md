# Working with AI on CollectionBuilder-CSV

This guide helps you collaborate effectively with AI assistants (Claude, ChatGPT, GitHub Copilot, etc.) when building CollectionBuilder-CSV sites.

## Understanding CollectionBuilder's Architecture

CollectionBuilder-CSV is a **data-driven** Jekyll framework. Nearly all customization happens through CSV and YAML configuration files, not by editing HTML or Liquid templates. Understanding this architecture is crucial for working effectively with AI.

### The Customization Hierarchy

Always work from the top down. AI assistants should check if a change can be made at a higher level before editing lower-level files:

1. **`_config.yml`** — Site identity, organization info, and the `metadata:` pointer
2. **`_data/theme.yml`** — Visual settings, feature toggles, navbar, map/timeline config
3. **`_data/config-*.csv`** — What fields appear on pages, browse cards, search, navigation
4. **`pages/*.md`** — Authored content using `_includes/feature/` includes
5. **`_sass/_custom.scss`** — CSS overrides (only when layers 1–4 cannot accomplish the goal)

### Critical Architecture Rules

When working with AI, emphasize these non-negotiable principles:

**Data-Driven Navigation**
- Never edit `_includes/collection-nav.html` to add/remove links
- Add rows to `_data/config-nav.csv` instead

**Data-Driven Item Pages**
- Item pages are auto-generated by `_plugins/cb_page_gen.rb` from your metadata CSV
- Never create `.html` files manually in `items/`
- Control field display via `_data/config-metadata.csv`, not by editing layout HTML

**Data-Driven Browse/Search**
- Configure browse cards and facets in `_data/config-browse.csv`
- Configure search indexing in `_data/config-search.csv`
- Don't edit `_layouts/browse.html` for field-level changes

**Custom Styling**
- Write all custom CSS in `_sass/_custom.scss`
- Never modify `_base.scss`, `_pages.scss`, or `_theme-colors.scss`

**Metadata Pointer**
- `_config.yml` sets `metadata: filename` with **no `.csv` extension**
- Access in Liquid: `site.data[site.metadata]` (bracket notation required)

## Key Configuration Files Reference

| File | Purpose | When to Edit |
|------|---------|-------------|
| `_config.yml` | Site identity, metadata CSV pointer | Setting up site, changing collections |
| `_data/theme.yml` | Visual/behavioral settings | Changing colors, fonts, page features |
| `_data/config-metadata.csv` | Item page field display | Adding/removing metadata fields on item pages |
| `_data/config-browse.csv` | Browse card fields, facets, sorting | Customizing browse page |
| `_data/config-search.csv` | Search index and results | Customizing what's searchable |
| `_data/config-nav.csv` | Navbar and footer links | Adding/removing navigation |
| `_data/config-map.csv` | Map popup fields | Customizing map display |
| `_data/<metadata>.csv` | Your collection items | Adding/editing collection content |
| `_sass/_custom.scss` | Custom styles | CSS that can't be done via `theme.yml` |

## Using Feature Includes for Content

CollectionBuilder provides powerful `_includes/feature/` includes for building rich content pages. AI assistants should always use these instead of raw HTML:

```liquid
{% include feature/image.html objectid="demo_001" width="75" %}
{% include feature/card.html header="Title" text="Body text" objectid="demo_004" %}
{% include feature/gallery.html filter-field="subject" filter-value="mining" %}
{% include feature/jumbotron.html objectid="demo_001" heading="My Heading" %}
{% include feature/alert.html text="Important!" color="warning" %}
{% include feature/button.html text="Browse" link="/browse.html" color="primary" %}
```

Full reference: `_includes/cb/feature_options.md` and `docs/markup.md`

## Common AI Pitfalls to Avoid

1. **Editing templates when config would work** — AI may try to edit `_includes/` or `_layouts/` when a CSV change would suffice
2. **Adding `.csv` to metadata pointer** — The `metadata:` value in `_config.yml` should never include the extension
3. **Using dot notation for metadata** — Must use `site.data[site.metadata]`, not `site.data.metadata`
4. **Manually creating item pages** — They're auto-generated; edit the metadata CSV instead
5. **Bootstrap 4 syntax** — This project uses Bootstrap 5 (`ms-`/`me-`, not `ml-`/`mr-`)
6. **Editing base SCSS files** — Custom styles belong in `_sass/_custom.scss` only

## Effective Prompts for AI

**Good prompts reference the architecture:**
- "Add a subject facet to the browse page" (AI will edit `config-browse.csv`)
- "Add a timeline item to the about page" (AI will use feature includes)
- "Change the navbar color to navy" (AI will edit `theme.yml`)

**Less effective prompts:**
- "Make the browse page show subjects" (unclear what "show" means)
- "Fix the navigation" (no context about what's broken)

## Build Commands

```bash
bundle exec jekyll serve    # Local dev server (localhost:4000)
bundle exec jekyll build    # Production build
JEKYLL_ENV=production bundle exec jekyll build  # Build with analytics/meta tags
```

## Tips for AI Collaboration

1. **Start with questions** — Ask AI to explain the architecture before making changes
2. **Request explanations** — Ask AI to explain why it's editing a particular file
3. **Review CSV changes carefully** — Small typos in config CSVs can break features
4. **Test incrementally** — Make one change at a time, rebuild, and verify
5. **Use existing docs** — Point AI to relevant `docs/` files for detailed information
